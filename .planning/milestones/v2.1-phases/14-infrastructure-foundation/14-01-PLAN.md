Inside `run()`, after creating `this.abortController`, compose signals. The SDK accepts `abortController: AbortController` (not AbortSignal), so use the event listener approach per research open question #1:
```typescript
// Thread external signal into internal controller
// Track whether the session query promise has settled
let sessionSettled = false;

if (signal) {
  if (signal.aborted) {
    clearTimeout(timeoutHandle);
    this.abortController = null;
    return {
      sessionId,
      status: 'cancelled' as const,
      toolCallCount: 0,
      duration: 0,
      finalResponse: '',
      error: 'Cancelled before start',
    };
  }
  signal.addEventListener('abort', () => {
    // Step 1 of grace period: signal abort to SDK
    this.abortController?.abort();

    // Step 2: 5-second grace period, then docker kill if session hasn't exited
    // Per locked decision: "signal abort to SDK, wait 5s for graceful shutdown, then docker kill"
    if (!sessionSettled) {
      setTimeout(async () => {
        if (!sessionSettled) {
          try {
            await execFileAsync('docker', ['kill', containerName], { timeout: 5000 });
          } catch {
            // Container may have already exited — ignore
          }
        }
      }, 5000);
    }
  }, { once: true });
}
```

Set `sessionSettled = true` in the `finally` block (before the existing docker kill fallback) so the grace period handler knows the session has exited and skips the redundant kill:
```typescript
} finally {
  sessionSettled = true;
  clearTimeout(timeoutHandle);
  // ... rest of existing finally block ...
}
```

NOTE: The existing `finally` block already calls `docker kill` unconditionally as a zombie-prevention fallback. This is fine — it runs when the session exits naturally or after SDK abort. The NEW grace period handler above is specifically for the case where the SDK abort doesn't cause the session to exit within 5 seconds (e.g., a hung process). The `sessionSettled` flag prevents double-kill.
