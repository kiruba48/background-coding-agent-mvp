### v3.0 Program Automator (In Progress)

**Milestone Goal:** Evolve the agent from a single-session chore automator into a program automator that can drive arbitrary **sweeping refactors** — any task shaped as "find all occurrences of X, transform to Y, verify Z" — across an entire repository. New task types must be addable as declarative recipes, not as new code phases.

**The shift in one sentence:** Stop treating a session as the unit of work. Make a **RefactorRun** the unit of work, and let sessions become replaceable workers against a persistent ledger, driven by a declarative recipe.

**What unlocks:**
- Language modernization (e.g., Java value types → records)
- Breaking upgrades with migration guides (e.g., Scio version bumps)
- UI component migrations (e.g., Backstage frontend swap)
- Schema-aware config edits (e.g., YAML/JSON fleet bumps)
- ...and any future task with the same shape — no new phase required.

**What explicitly does NOT ship in v3.0:**
- Full language-to-language rewrites (Java → Kotlin). Different shape, needs equivalence-testing infra. Defer to v4.
- Multi-repo fleet orchestration. One repo per run; wrap externally if needed.
- Live-environment verification (prod-shaped DBs, live services). Defer.
- Creative/subjective tasks (visual polish, "make this nicer"). Not a sweeping refactor.

- [ ] **Phase 28: Sweeping-Refactor Task Type + Discovery Pass** — New intent category, read-only discovery pass producing a typed target list
- [ ] **Phase 29: RefactorRun Orchestrator** — Multi-session state, long-lived shared worktree/branch, per-chunk worker loop
- [ ] **Phase 30: Differential Verification** — Baseline snapshot at run start; verify each chunk against baseline instead of absolute green
- [ ] **Phase 31: Context Bundle Mount + Judge Scope Injection** — Read-only `/context/` mount; judge reads recipe spec as authoritative scope
- [ ] **Phase 32: Recipe Format + Recipe Runner** — Declarative YAML recipe schema; generic runner that executes any recipe
- [ ] **Phase 33: Capability Toolbox** — Shared, task-agnostic MCP tools: `config_edit`, `ast_query`/`ast_rewrite`, `import_rewrite`, `rewrite_run`, `test_baseline`/`test_compare`, `doc_retrieve`
- [ ] **Phase 34: Conversational Recipe Authoring** — Extend Phase 22's scoping dialogue to emit recipe drafts from a four-question interview

## Phase Details

### Phase 28: Sweeping-Refactor Task Type + Discovery Pass
**Goal**: The agent accepts a new `sweeping-refactor` task type and, given a target description, produces a structured target list (file + location + kind) as a read-only first step — no edits, no PR.
**Depends on**: Phase 27 (reuses investigation-mode plumbing for read-only execution)
**Requirements**: REFAC-01, REFAC-02, REFAC-03 (to be enumerated in v3.0-REQUIREMENTS.md)
**Success Criteria** (what must be TRUE):
  1. The intent parser routes phrases like "modernize all POJOs to records" or "migrate YAML image tags" to the `sweeping-refactor` task type without a type flag
  2. Running discovery on a repo writes a `targets.json` to the state store containing an array of `{file, locator, kind}` entries
  3. A discovery pass cannot write or edit files — enforced by the existing PreToolUse Write/Edit block from Phase 27
  4. Discovery output is deterministic given the same repo, recipe discovery block, and tool versions
**Plans**: 2 plans

Plans:
- [ ] 28-01-PLAN.md — Intent parser `sweeping-refactor` category, fast-path verbs, LLM parser extension
- [ ] 28-02-PLAN.md — Discovery prompt builder, structured `targets.json` output, state store persistence

### Phase 29: RefactorRun Orchestrator
**Goal**: A `RefactorRun` entity persists across sessions — it owns a long-lived worktree and branch, a target ledger (pending/in-progress/done/failed/skipped), and drives per-chunk sessions until the ledger is exhausted.
**Depends on**: Phase 28 (consumes `targets.json`)
**Requirements**: REFAC-04, REFAC-05, REFAC-06, REFAC-07
**Success Criteria** (what must be TRUE):
  1. Starting a refactor run creates one worktree and one branch that persists across multiple sessions — `WorktreeManager` gains a "reuse existing" path and does not prune the run's worktree between sessions
  2. The orchestrator pops the next pending target, spawns a scoped session for that chunk, and on success marks the target `done` and records the commit SHA on the shared branch
  3. A chunk failure marks the target `failed` with a reason and continues to the next pending target — the run does not abort on a single failure
  4. `agent refactor resume <run-id>` picks up exactly where the run left off, including after a process crash
  5. Per-task-type session envelope config is honored: a sweeping-refactor chunk can exceed the default 10-turn/5-minute limits (configurable, bounded)
**Plans**: 3 plans

Plans:
- [ ] 29-01-PLAN.md — `RefactorRun` data model, state store schema, `WorktreeManager` reuse path
- [ ] 29-02-PLAN.md — Worker loop: pop → spawn session → commit → mark ledger → repeat
- [ ] 29-03-PLAN.md — `agent refactor resume`, crash recovery, per-task-type envelope config

### Phase 30: Differential Verification
**Goal**: Instead of requiring an absolute green build/test/lint result, the composite verifier compares against a **baseline snapshot** captured at run start — "same tests pass, no new lint classes, build still green" — so long-running refactors can't silently regress chunks they already finished.
**Depends on**: Phase 29 (baseline lives on the RefactorRun)
**Requirements**: REFAC-08, REFAC-09, REFAC-10
**Success Criteria** (what must be TRUE):
  1. Starting a refactor run captures a baseline record: build result, full test result set (name → pass/fail), lint warning counts by class
  2. After each chunk, the composite verifier fails the chunk if any previously-passing test now fails, if a new lint class appears, or if the build regresses — even when the chunk's own edits look fine in isolation
  3. Recipes may declare invariants (e.g., "no new public API surface") that the verifier enforces alongside the baseline comparison
  4. A chunk is allowed to add new tests; dropping a pre-existing test is blocked unless the recipe explicitly permits it
**Plans**: 2 plans

Plans:
- [ ] 30-01-PLAN.md — Baseline capture at run start, baseline storage on `RefactorRun`
- [ ] 30-02-PLAN.md — Differential comparison in composite verifier, invariant check hooks

### Phase 31: Context Bundle Mount + Judge Scope Injection
**Goal**: The sandbox can mount an external read-only `/context/` directory (migration guides, API maps, schemas, component maps) that the agent is told about, and the LLM Judge receives the recipe's scope definition as the authoritative "what is in scope" signal — so it stops vetoing legitimate mechanical changes.
**Depends on**: Phase 29
**Requirements**: REFAC-11, REFAC-12, REFAC-13
**Success Criteria** (what must be TRUE):
  1. `agent refactor start --context-bundle ./guide/` mounts the bundle at `/context/` with `:ro` inside the sandbox; the agent prompt references its availability
  2. The context bundle cannot be written by the agent — enforced at Docker mount level and PreToolUse hook level
  3. The LLM Judge prompt includes the recipe's transformation spec as the scope definition; diffs matching the spec are not vetoed as "scope creep" even when large
  4. A run without a context bundle behaves identically to today (no regression on existing task types)
**Plans**: 2 plans

Plans:
- [ ] 31-01-PLAN.md — ContainerManager context-bundle mount, PreToolUse hook coverage, prompt plumbing
- [ ] 31-02-PLAN.md — Judge prompt extension: recipe-driven scope, vetoed-diff regression test

### Phase 32: Recipe Format + Recipe Runner
**Goal**: A refactor run is fully specified by a declarative YAML recipe with four slots — discovery, transformation, verification, context. A generic runner reads the recipe and executes it using the infrastructure from Phases 28-31. **Adding a new task type is writing a recipe, not writing code.**
**Depends on**: Phases 28, 29, 30, 31
**Requirements**: REFAC-14, REFAC-15, REFAC-16, REFAC-17, REFAC-18
**Success Criteria** (what must be TRUE):
  1. A recipe YAML validates against the v3.0 recipe schema (see Appendix A); invalid recipes are rejected before run start with a clear error
  2. The runner supports all three transformation strategies — `deterministic`, `end-state-prompt`, `doc-grounded` — and routes each to the correct execution path
  3. Three reference recipes ship with v3.0 and run end-to-end against fixture repos: one `deterministic` (YAML config bump), one `end-state-prompt` (mechanical code rewrite), one `doc-grounded` (breaking upgrade with migration guide in context bundle)
  4. Recipe execution is reproducible: same repo + same recipe + same tool versions produces the same ledger outcome (modulo LLM non-determinism for prompt-driven chunks)
  5. A new task type can be added by authoring a recipe file alone — no changes to the runner, orchestrator, or intent parser
**Plans**: 3 plans

Plans:
- [ ] 32-01-PLAN.md — Recipe schema definition, YAML loader, validator, error reporting
- [ ] 32-02-PLAN.md — Recipe runner: slot execution (discovery → transform → verify), strategy dispatch
- [ ] 32-03-PLAN.md — Reference recipes (config bump, code rewrite, doc-grounded upgrade) + fixture repos

### Phase 33: Capability Toolbox
**Goal**: A small, **task-agnostic** set of reusable MCP tools that recipes call as primitives. Not plugins. Not task-specific. Each is broadly useful across many recipes; adding a new tool is a small PR, not a roadmap phase.
**Depends on**: Phase 32 (recipes reference tools by name)
**Requirements**: REFAC-19, REFAC-20, REFAC-21
**Success Criteria** (what must be TRUE):
  1. The following MCP tools exist, are exposed only to sweeping-refactor task types, and have unit + integration tests:
     - `config_edit` — roundtrip-safe YAML/JSON edits with optional JSON-Schema validation (preserves comments, formatting)
     - `ast_query` / `ast_rewrite` — tree-sitter-backed structural search and edit across any supported language
     - `import_rewrite` — rename/replace imports and symbols across a language's module system
     - `rewrite_run` — invoke a pre-registered OpenRewrite / jscodeshift / semgrep recipe by id (toolchain pre-seeded in the sandbox image, no network required)
     - `test_baseline` / `test_compare` — capture and diff test result sets for differential verification
     - `doc_retrieve` — BM25 lookup over the mounted context bundle (no embeddings, no network)
  2. Each tool is usable independently from recipes (can be called from a generic session) — tools are general-purpose, not recipe-coupled
  3. The three reference recipes in Phase 32 together exercise every tool at least once
**Plans**: 3 plans

Plans:
- [ ] 33-01-PLAN.md — `config_edit` (yaml roundtrip + jsonc-parser + schema validation) and `doc_retrieve` (BM25)
- [ ] 33-02-PLAN.md — `ast_query` / `ast_rewrite` via tree-sitter, `import_rewrite`
- [ ] 33-03-PLAN.md — `rewrite_run` (OpenRewrite/jscodeshift/semgrep bridge, pre-seeded image), `test_baseline` / `test_compare`

### Phase 34: Conversational Recipe Authoring
**Goal**: Users don't hand-write YAML. When the scoping dialogue (from Phase 22) detects a sweeping-refactor shape, it asks four questions — *what marks a target site, what should it become, how do we know it still works, any docs I should read?* — and emits a recipe draft the user confirms in Slack or REPL.
**Depends on**: Phase 32 (emits valid recipes), Phase 22 (scoping dialogue)
**Requirements**: REFAC-22, REFAC-23, REFAC-24
**Success Criteria** (what must be TRUE):
  1. In the REPL and in Slack, a user phrase matching a sweeping-refactor shape triggers the four-question interview instead of the normal scoping dialogue
  2. The interview produces a recipe that validates against the Phase 32 schema without manual edits
  3. The user sees a human-readable summary of the recipe (not raw YAML) before confirming; advanced users can `/edit recipe` to see and tweak the YAML
  4. A confirmed recipe immediately starts a RefactorRun; the user can monitor progress with `agent refactor status <run-id>`
**Plans**: 2 plans

Plans:
- [ ] 34-01-PLAN.md — Four-question interview logic, recipe draft emission, human-readable summary
- [ ] 34-02-PLAN.md — REPL + Slack wiring, `agent refactor status` command, edit path

## Appendix A — Recipe Schema Spec (v3.0)

A refactor recipe is a YAML document with exactly four top-level slots plus metadata. The runner in Phase 32 validates against this schema before starting a run.

### Top-level structure

```yaml
# Metadata
name: string                      # required, unique, kebab-case
version: string                   # required, semver — the recipe's own version
description: string               # required, one-line human summary
authors: [string]                 # optional

# The four slots
discovery: <DiscoveryBlock>       # required
transformation: <TransformBlock>  # required
verification: <VerifyBlock>       # required
context: <ContextBlock>           # optional (null if no external docs needed)

# Execution tuning
chunking: <ChunkingBlock>         # optional, defaults applied
envelope: <EnvelopeBlock>         # optional, defaults applied
```

### DiscoveryBlock

Produces a list of target sites. Exactly one `tool` per recipe.

```yaml
discovery:
  tool: grep | ast_query | config_query | import_scan | custom
  # --- grep ---
  patterns: [string]              # regex or literal, ripgrep-compatible
  globs: [string]                 # optional file filter
  # --- ast_query ---
  language: java | python | typescript | ...
  query: string                   # tree-sitter query or a named preset
  # --- config_query ---
  files: [string]                 # glob
  path: string                    # JSONPath-style locator
  # --- import_scan ---
  module: string                  # e.g. "com.spotify.scio.v1"
  symbols: [string]               # optional specific symbols
  # --- custom ---
  prompt: string                  # end-state prompt for a discovery-only agent session
                                  # (falls back to LLM-driven discovery)
```

All variants produce the same output shape:

```json
[{"file": "path/to/x", "locator": "line:42 | jsonpath:$.a.b | symbol:Foo", "kind": "string"}]
```

### TransformBlock

Defines how each target site is changed. Exactly one `strategy`.

```yaml
transformation:
  strategy: deterministic | end-state-prompt | doc-grounded

  # --- deterministic ---
  # Calls a capability tool directly. No LLM per chunk. Fastest & safest.
  capability: config_edit | ast_rewrite | import_rewrite | rewrite_run
  op: set | delete | replace | run
  args: <map>                     # tool-specific arguments

  # --- end-state-prompt ---
  # Hands the spec to the agent with scoped tool access. Your existing generic path.
  spec: string                    # end-state description, NOT step-by-step
  allowed_tools: [string]         # subset of the normal allowlist
  capability_hints: [string]      # optional hints like "rewrite_run:<recipe-id>"

  # --- doc-grounded ---
  # Same as end-state-prompt but REQUIRES the agent to consult the context bundle
  # via doc_retrieve before editing each chunk.
  spec: string
  retrieval_query_template: string  # e.g. "how to migrate {symbol}"
  allowed_tools: [string]
```

### VerifyBlock

Defines what "correct" means for this recipe. Runs after every chunk, against the baseline.

```yaml
verification:
  baseline: [build, test, lint, schema_validate, formatter_diff]
                                  # which checks to capture at run start
  invariants:                     # optional, recipe-specific assertions
    - "all pre-existing tests pass with same results"
    - "no new public API surface"
    - "formatter produces identical output on untouched files"
  schema: string                  # optional path, required if schema_validate is used
  test_filter: string             # optional, limit baseline to a test subset
  allow_dropped_tests: boolean    # default false — blocks removing a pre-existing test
  chunk_scope:
    must_touch: [string]          # globs the chunk's diff MUST be within
    must_not_touch: [string]      # globs the chunk's diff MUST NOT touch
```

### ContextBlock

Optional external reference material mounted read-only at `/context/` in the sandbox.

```yaml
context:
  bundle: string | null           # path to a directory; null = no context
  retrieval: none | bm25          # default bm25 when bundle is set
  index:                          # optional custom index hints
    include: [string]             # globs within bundle
    exclude: [string]
```

### ChunkingBlock

```yaml
chunking:
  strategy: per-file | per-target | per-module | grouped
  group_by: string                # required when strategy=grouped (e.g. "directory")
  max_parallel: integer           # default 1; >1 requires worktree sharding (v3.1+)
  max_chunk_size: integer         # hard cap on targets per chunk
```

### EnvelopeBlock

Per-chunk resource limits, overriding global defaults.

```yaml
envelope:
  max_turns: integer              # default 20 for sweeping-refactor (vs 10 generic)
  timeout_seconds: integer        # default 900 (15 min) for sweeping-refactor
  max_retries: integer            # default 3
```

### Example: Java value types → records

```yaml
name: java-value-types-to-records
version: 1.0.0
description: Replace eligible POJOs with Java records, preserving public API.

discovery:
  tool: ast_query
  language: java
  query: "classes with only private final fields, getters, equals, hashCode"

transformation:
  strategy: end-state-prompt
  spec: |
    Convert eligible POJOs to Java records. Preserve the public API exactly —
    method names, return types, and nullability contracts. Leave classes that
    have setters, inheritance, or non-trivial constructors untouched.
  capability_hints: ["rewrite_run:org.openrewrite.java.migrate.lang.ClassToRecord"]
  allowed_tools: [read_file, ast_query, ast_rewrite, rewrite_run, git]

verification:
  baseline: [build, test]
  invariants:
    - "all pre-existing tests pass with same results"
    - "no new public API surface"
  chunk_scope:
    must_touch: ["**/*.java"]
    must_not_touch: ["**/pom.xml", "**/build.gradle*"]

context: null

chunking:
  strategy: per-file
  max_parallel: 1

envelope:
  max_turns: 25
  timeout_seconds: 900
```

```yaml
chunking:
  strategy: per-module
  max_parallel: 1

  max_turns: 30
  timeout_seconds: 1200
```

### Example: YAML image tag bump (deterministic)

```yaml
name: helm-image-tag-bump
version: 1.0.0
description: Update image.tag across all values.yaml files to a new version.

discovery:
  tool: config_query
  files: ["**/values.yaml"]
  path: "image.tag"

transformation:
  strategy: deterministic
```

### Design notes

- **Exactly four slots.** Resist adding a fifth. Every new field should land inside one of the existing slots or be rejected as out-of-scope.
- **Strategy-dispatch, not inheritance.** The runner switches on `transformation.strategy` and routes to one of three code paths. No class hierarchy, no plugin registry for transformations.
- **Tools are the extension point, not recipes.** When a recipe can't be expressed with existing capability tools, add a tool (Phase 33 cadence), not a new recipe type.
- **Determinism first.** Prefer `deterministic` over `end-state-prompt` over `doc-grounded` whenever the task shape allows it — faster, cheaper, no LLM veto risk.
- **Recipes version themselves.** A recipe's `version` field lets the state store tie past runs to exact recipe contents. Changing a recipe is not a silent rewrite of history.
