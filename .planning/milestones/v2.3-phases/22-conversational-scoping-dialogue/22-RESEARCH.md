## User Constraints (from CONTEXT.md)

### Locked Decisions

**Question generation:**
- Questions are dynamically generated by the LLM, not fixed templates
- Generated in the same Haiku 4.5 structured output call as intent parsing — no additional LLM call
- Extend `ResolvedIntent` with `scopingQuestions?: string[]` field on the response schema
- LLM receives: task description, repo file tree (top-level), manifest dependencies, and task history as context
- LLM decides 0-3 questions based on task ambiguity — clear tasks like "rename X to Y" may get 0 questions
- Questions are specific to the task and repo, not generic templates

**Answer format and SCOPE block:**
- Answers merge into a new `SCOPE HINTS (from user):` section appended after the existing SCOPE fence in buildGenericPrompt()
- Each Q&A pair rendered as `- [question]: [answer]`
- Skipped questions (user presses Enter) are omitted entirely from SCOPE HINTS and confirm display
- If all questions skipped or 0 questions generated, no SCOPE HINTS section appears

**Confirm display (SCOPE-04):**
- Scope hints shown inline under the intent display block, before the `Proceed? [Y/n]` prompt
- Indented bullet list: `Scope hints:` header followed by `  - [hint]` per answered question
- No separate confirmation step for scope — single Y/n covers intent + scope

**Pipeline placement:**
- Flow: parse → clarify → scope → confirm → run
- Scoping runs after parse/clarify, before confirm
- Scoping only runs on confident parses — if clarification was needed, the re-parsed intent may or may not include questions (LLM decides on re-parse)
- 0 questions → skip silently to confirm (user never sees scoping step)

**Abort behavior:**
- Ctrl+C during scoping cancels remaining questions
- Answers collected so far are kept
- Flow continues to confirm with whatever hints were captured

**Adapter contract:**
- Add `askQuestion?: (prompt: string) => Promise<string | null>` to SessionCallbacks
- Optional method — adapters that don't implement it skip scoping entirely
- Session core calls askQuestion in a loop, once per generated question
- Return `null` = skip that question (Enter key or adapter choosing to skip)
- CLI adapter implements via existing readline `askQuestion()` helper in repl.ts
- Slack adapter: does not implement askQuestion in v2.3 — relies on existing clarify for ambiguity

**Bypass rules (SCOPE-05):**
- Scoping only triggers for `generic` taskType
- Dependency updates (maven, npm) receive no scoping questions — already fully parameterized
- If `SessionCallbacks.askQuestion` is undefined, scoping is silently bypassed

### Claude's Discretion
- Exact system prompt wording for scoping question generation
- How repo file tree and manifest deps are formatted in the LLM context
- Whether to include question labels/categories or just plain question text
- Fallback if scoping questions field is missing from LLM response (treat as 0 questions)

### Deferred Ideas (OUT OF SCOPE)
- **SLCK-10**: Slack scoping via Block Kit modals — deferred to future milestone
- **SCOPE-06 enhancement**: LLM-generated questions could include suggested answers (autocomplete-style) in future iterations

## Summary

Phase 22 adds a conversational scoping dialogue step to the REPL pipeline for generic tasks. When a user submits a generic task, the intent parser (Haiku 4.5) now returns 0-3 dynamically generated scoping questions alongside the parsed intent. Before reaching the confirm step, the session core calls `askQuestion` for each question; user answers are collected as "scope hints" and injected into `buildGenericPrompt()` as a `SCOPE HINTS (from user):` block. The scope hints are also rendered at the confirm step so the user can review them before proceeding.

The change touches five files: `src/intent/types.ts` (schema extension), `src/intent/llm-parser.ts` (OUTPUT_SCHEMA + Zod schema), `src/repl/types.ts` (SessionCallbacks), `src/repl/session.ts` (pipeline insertion), `src/prompts/generic.ts` (SCOPE HINTS rendering), `src/intent/confirm-loop.ts` (displayIntent extension), and `src/cli/commands/repl.ts` (CLI askQuestion callback + confirm display). The changes are purely additive — existing behaviour for dependency updates and for adapters that omit `askQuestion` is fully preserved via the optional callback pattern.

The primary challenge is extending the Haiku 4.5 structured output schema without breaking the existing `IntentSchema` Zod validation, and threading `scopeHints` through `processInput()` to both `buildGenericPrompt` and `displayIntent` without introducing coupling between session core and prompt logic.

**Primary recommendation:** Implement `runScopingDialogue()` as a pure function in `src/repl/session.ts`, extend `IntentSchema`/`OUTPUT_SCHEMA` with `scopingQuestions` field, and pass `scopeHints` as a parameter through the existing function signatures rather than storing on state.

## Architecture Patterns

### Pipeline Threading Pattern

The existing `processInput()` pipeline uses a linear flow where each step returns data that the next step consumes. The scoping dialogue follows this same pattern — `runScopingDialogue()` is a pure async function that receives the intent and callbacks and returns `string[]` (the collected hints).

```
parse → clarify → [runScopingDialogue] → confirm(scopeHints) → runAgent(scopeHints)
```

`scopeHints` is threaded as a local variable in `processInput()`, passed to `callbacks.confirm` via the existing closure (CLI adapter's `confirmCb` already has access to intent), and passed to `buildGenericPrompt` inside `runAgent` setup.

### Extend IntentSchema and OUTPUT_SCHEMA (llm-parser.ts)

**What:** Add `scopingQuestions` to both the Zod schema and the JSON schema object passed to the API.

**Key constraint:** The field is optional on `IntentResult` (Zod `.optional()` or with default `[]`) so existing callers don't break. The LLM system prompt must instruct when to populate it — only for `taskType: generic` with ambiguous scope.

```typescript
// src/intent/types.ts — IntentSchema extension
export const IntentSchema = z.object({
  // ... existing fields unchanged ...
  scopingQuestions: z.array(z.string()).optional().default([]),
});

// src/intent/llm-parser.ts — OUTPUT_SCHEMA extension
scopingQuestions: {
  type: 'array',
  items: { type: 'string' },
},
// Add 'scopingQuestions' to required[] array
```

**Fallback:** If the LLM omits `scopingQuestions` or returns null, Zod `.default([])` ensures an empty array — no questions shown.

### runScopingDialogue() Pure Function

```typescript
// src/repl/session.ts
export async function runScopingDialogue(
  questions: string[],
  askQuestion: (prompt: string) => Promise<string | null>,
): Promise<string[]> {
  const hints: string[] = [];
  for (const q of questions.slice(0, 3)) {
    const answer = await askQuestion(`  ${q} `);
    if (answer !== null && answer.trim() !== '') {
      hints.push(`${q}: ${answer.trim()}`);
    }
    // null (Ctrl+C) or empty string (Enter) → skip, continue loop
  }
  return hints;
}
```

**Note:** Ctrl+C inside `askQuestion` returns `null` (per existing AbortController pattern in repl.ts line 152-163). The loop `continue`s on null, collecting whatever hints came before Ctrl+C. This satisfies the abort-keeps-partial-answers constraint.

### processInput() Integration Point

In `session.ts`, after the clarify block (Step 2) and before the confirm block (Step 3):

```typescript
// Step 2.5: Scoping dialogue (generic tasks only, if adapter implements askQuestion)
let scopeHints: string[] = [];
if (
  intent.taskType === 'generic' &&
  intent.scopingQuestions &&
  intent.scopingQuestions.length > 0 &&
  callbacks.askQuestion
) {
  scopeHints = await runScopingDialogue(intent.scopingQuestions, callbacks.askQuestion);
}

// Step 3: Confirm loop via callback — pass scopeHints for display
const confirmed = await callbacks.confirm(intent, reparse, scopeHints);
```

**Confirmed signature change:** `confirm` callback must accept optional `scopeHints` parameter — OR the CLI adapter's `confirmCb` closes over a `scopeHints` variable set before calling confirm. The latter avoids changing `SessionCallbacks['confirm']` signature, keeping the interface stable.

**Recommended approach:** Use a closure variable in processInput and pass it to `displayIntent` from within the CLI adapter's `confirmCb`. This means `processInput` does NOT change the `confirm` signature — instead it stores `scopeHints` in a local, and passes it to `buildGenericPrompt` when constructing `agentOptions`. The CLI adapter's `confirmCb` is a closure, so it can't receive `scopeHints` unless `processInput` threads it. Two clean approaches:

**Option A (simpler):** Change `confirm` signature to accept `scopeHints?: string[]`. Planner should choose based on minimal interface disruption.

**Option B (no signature change):** `processInput` uses `scopeHints` purely internally — passes it to `buildGenericPrompt` for the agent run, and the confirm callback is enhanced by having processInput call `displayIntent(intent, scopeHints)` before calling `callbacks.confirm`. This removes scope display from the CLI adapter — session core handles it.

Option B is slightly cleaner architecturally (session core owns the pipeline order) but requires session core to import `displayIntent`. Option A keeps confirm callback fully responsible for display. The planner should choose; both are valid.

### CLI askQuestion Callback Implementation

The existing `askQuestion()` helper in `repl.ts` (lines 152-163) already handles AbortController and returns `null` on abort. The CLI callback implementation is:

```typescript
// src/cli/commands/repl.ts — in callbacks object
askQuestion: async (prompt: string) => {
  return askQuestion(rl, prompt, activeQuestionControllerRef);
},
```

This reuses the exact existing helper signature with no new code beyond wiring it into the `callbacks` object literal.

### Anti-Patterns to Avoid

- **Adding scopingQuestions to ResolvedIntent:** The field belongs on `IntentResult` (raw LLM output) only. `ResolvedIntent` should also carry it forward so session core can access it without re-calling the LLM, but make it `scopingQuestions?: string[]` optional so it doesn't affect dep-update intents.
- **A separate LLM call for questions:** Locked decision — questions come from the same Haiku 4.5 structured output call. Never add a second API call.
- **Blocking confirm on empty hints:** If `scopeHints.length === 0`, skip the hints display block entirely — don't render an empty `Scope hints:` section.
- **Storing scopeHints on ReplState:** Scope hints are per-invocation, not session state. Keep them as local variables in `processInput()`.
- **Adding scopingQuestions to the fast-path result:** Fast-path returns `FastPathResult` for dep updates only. Generic tasks always go through LLM. No change needed in `fast-path.ts`.

## Common Pitfalls

### Pitfall 1: Zod Schema vs JSON OUTPUT_SCHEMA Drift

**What goes wrong:** Adding `scopingQuestions` to `IntentSchema` (Zod) but forgetting to add it to `OUTPUT_SCHEMA` (plain JSON object passed to the API), or vice versa. The LLM receives only `OUTPUT_SCHEMA` — if the field is missing there, the LLM will never output it. If missing from Zod, the parse fails even when the LLM does output it.

**How to avoid:** Update both `IntentSchema` in `types.ts` AND `OUTPUT_SCHEMA` in `llm-parser.ts`. Add `scopingQuestions` to the `required` array in `OUTPUT_SCHEMA` (or leave out of required and let Zod `.default([])` handle missing values — the latter is safer).

**Warning signs:** Tests where `scopingQuestions` is always undefined despite LLM response containing it; or LlmParseError on valid responses.

### Pitfall 2: confirm() Signature Change Breaking Existing Tests

**What goes wrong:** Changing `SessionCallbacks['confirm']` to accept `scopeHints` as a third parameter breaks `session.test.ts` and `repl.test.ts` where `confirm` is a vi.fn() mock with specific arity expectations.

**How to avoid:** If Option A (signature change) is chosen, update all `makeCallbacks()` test helpers to include the new parameter. Alternatively choose Option B (closure approach) to avoid the signature change.

### Pitfall 3: scopingQuestions on Fast-Path Generic Tasks

**What goes wrong:** A generic task may be parsed via fast-path for dep updates — but generic tasks are never fast-pathed. Still, if a generic task's `ResolvedIntent` is constructed in index.ts from `llmPreResult`, the `scopingQuestions` field must be forwarded from `llmPreResult` to `ResolvedIntent`.

**How to avoid:** In `src/intent/index.ts`, when constructing the return value for generic tasks, include:
```typescript
scopingQuestions: isGeneric ? llmResult.scopingQuestions : undefined,
```

### Pitfall 4: Empty-String vs Null Distinction in askQuestion

**What goes wrong:** The existing `askQuestion()` helper returns the string value of rl.question() — which returns `''` (empty string) for Enter, not `null`. The `null` case only occurs on AbortError. If `runScopingDialogue` only checks `answer !== null` and not `answer.trim() !== ''`, empty Enter presses will be treated as answers with blank content.

**How to avoid:** `runScopingDialogue` must check both conditions:
```typescript
if (answer !== null && answer.trim() !== '') { /* add hint */ }
```

### Pitfall 5: LLM Generates Questions for Dep Updates

**What goes wrong:** System prompt instructs LLM to generate scoping questions for all tasks, including dep updates. Questions would be generated but bypassed in the pipeline — wasteful token spend.

**How to avoid:** Update the system prompt to include: "Generate `scopingQuestions` only when `taskType` is `generic`. Set `scopingQuestions` to `[]` for all dependency update tasks." This is within Claude's Discretion (exact system prompt wording).

### Pitfall 6: max_tokens Too Low for scopingQuestions

**What goes wrong:** Current `max_tokens: 512` in llmParse. Three question strings of ~80 chars each add ~240 characters to the response. This is within the 512 limit, but borderline for edge cases with long questions.

**How to avoid:** Increase `max_tokens` to `768` or `1024` in the Haiku 4.5 call to accommodate the additional fields. The cost increase is negligible (Haiku is extremely cheap).

## Code Examples

### Existing askQuestion Helper (reuse as-is)

```typescript
// src/cli/commands/repl.ts lines 152-163 — SOURCE OF TRUTH
async function askQuestion(
  rl: Interface,
  prompt: string,
  activeQuestionControllerRef: { current: AbortController | null },
): Promise<string | null> {
  const ctrl = new AbortController();
  activeQuestionControllerRef.current = ctrl;
  try {
    return await rl.question(prompt, { signal: ctrl.signal });
  } catch (err) {
    if ((err as Error).name === 'AbortError') return null;
    throw err;
  } finally {
    activeQuestionControllerRef.current = null;
  }
}
```

## Open Questions

1. **confirm() signature: Option A (add `scopeHints` param) vs Option B (closure/session-core-calls-displayIntent)**
   - What we know: Option A changes the SessionCallbacks interface and all test mocks; Option B avoids interface change but requires session core to import displayIntent
   - Recommendation: The planner should choose Option A for cleaner data flow — the Slack adapter simply ignores the parameter, and test mocks use `vi.fn()` which accepts any signature

2. **System prompt instruction for scoping questions: how to phrase the repo file tree context**
   - What we know: CONTEXT.md says "LLM receives top-level file tree" — but `readManifestDeps()` currently only reads manifest deps, not directory tree
   - What's unclear: Whether to add a `readTopLevelTree()` function to context-scanner.ts, or pass the tree as a separate parameter
   - Recommendation: Add a lightweight `readTopLevelDirs()` helper that reads `readdirSync(repoPath)` (one level, no recursion) and appends to manifestContext — this is Plan 01 scope

3. **`scopingQuestions` in `required[]` of OUTPUT_SCHEMA**
   - Putting it in `required` means old cached intent responses that don't include the field will fail Zod validation
   - Recommendation: Keep out of `required` (LLM will still output it due to prompt instruction) and rely on Zod `.default([])` for the fallback
