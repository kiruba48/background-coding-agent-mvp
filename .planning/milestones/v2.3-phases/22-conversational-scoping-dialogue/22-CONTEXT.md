# Phase 22: Conversational Scoping Dialogue - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Dynamic, LLM-generated scoping questions asked before the confirm step for generic tasks. The intent parser (Haiku 4.5) generates 0-3 context-aware scoping questions in the same call that parses intent. Answers merge into a SCOPE HINTS block in the agent prompt. Dependency updates bypass scoping entirely. Includes SCOPE-06 (pulled forward from future requirements).

</domain>

<decisions>
## Implementation Decisions

### Question generation
- Questions are dynamically generated by the LLM, not fixed templates
- Generated in the same Haiku 4.5 structured output call as intent parsing — no additional LLM call
- Extend `ResolvedIntent` with `scopingQuestions?: string[]` field on the response schema
- LLM receives: task description, repo file tree (top-level), manifest dependencies, and task history as context
- LLM decides 0-3 questions based on task ambiguity — clear tasks like "rename X to Y" may get 0 questions
- Questions are specific to the task and repo, not generic templates

### Answer format & SCOPE block
- Answers merge into a new `SCOPE HINTS (from user):` section appended after the existing SCOPE fence in buildGenericPrompt()
- Each Q&A pair rendered as `- [question]: [answer]`
- Skipped questions (user presses Enter) are omitted entirely from SCOPE HINTS and confirm display
- If all questions skipped or 0 questions generated, no SCOPE HINTS section appears

### Confirm display (SCOPE-04)
- Scope hints shown inline under the intent display block, before the `Proceed? [Y/n]` prompt
- Indented bullet list: `Scope hints:` header followed by `  - [hint]` per answered question
- No separate confirmation step for scope — single Y/n covers intent + scope

### Pipeline placement
- Flow: parse → clarify → **scope** → confirm → run
- Scoping runs after parse/clarify, before confirm
- Scoping only runs on confident parses — if clarification was needed, the re-parsed intent may or may not include questions (LLM decides on re-parse)
- 0 questions → skip silently to confirm (user never sees scoping step)

### Abort behavior
- Ctrl+C during scoping cancels remaining questions
- Answers collected so far are kept
- Flow continues to confirm with whatever hints were captured

### Adapter contract
- Add `askQuestion?: (prompt: string) => Promise<string | null>` to SessionCallbacks
- Optional method — adapters that don't implement it skip scoping entirely
- Session core calls askQuestion in a loop, once per generated question
- Return `null` = skip that question (Enter key or adapter choosing to skip)
- CLI adapter implements via existing readline `askQuestion()` helper in repl.ts
- Slack adapter: does not implement askQuestion in v2.3 — relies on existing clarify (Block Kit buttons) for ambiguity. SLCK-10 (Slack scoping via modals) stays deferred.

### Bypass rules (SCOPE-05)
- Scoping only triggers for `generic` taskType
- Dependency updates (maven, npm) receive no scoping questions — already fully parameterized
- If `SessionCallbacks.askQuestion` is undefined, scoping is silently bypassed

### Claude's Discretion
- Exact system prompt wording for scoping question generation
- How repo file tree and manifest deps are formatted in the LLM context
- Whether to include question labels/categories or just plain question text
- Fallback if scoping questions field is missing from LLM response (treat as 0 questions)

</decisions>

<specifics>
## Specific Ideas

- Questions should feel natural and task-specific, not like a form. E.g., "This repo has both `src/auth/` and `src/middleware/` — which area should the error handling focus on?" rather than "Which files?"
- The intent parser already has all the context (task + repo) — extending its response is the lowest-latency path to dynamic questions

</specifics>
